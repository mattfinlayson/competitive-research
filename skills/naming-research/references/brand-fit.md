# Brand Architecture Fit Reference

Product evolution and naming scalability.

## Overview

A name should support product evolution beyond the initial use case. This reference helps evaluate whether a name can grow with your product.

## The Three-Phase Framework

### Phase 1: Initial Product

**Focus**: Core use case, clear value proposition

Examples:
- GDPR sub-processor compliance
- Single feature SaaS
- MVP product

**Naming needs**:
- Clear core positioning
- Single use case clarity
- Initial market fit

### Phase 2: Expansion

**Focus**: Additional features, adjacent use cases

Examples:
- GDPR + SOC 2 compliance
- Multiple feature areas
- Enterprise features

**Naming needs**:
- Scope flexibility
- Feature extension capability
- Market breadth

### Phase 3: Platform

**Focus**: Ecosystem, third-party integration

Examples:
- Compliance platform
- Multi-product suite
- Marketplace

**Naming needs**:
- Platform positioning
- Ecosystem fit
- Partner integration

## Brand Architecture Patterns

### 1. House of Brands

Multiple distinct brands under one company.

**Examples**:
- P&G: Tide, Pampers, Gillette
- Google: Search, Maps, Android (somewhat)

**When to use**:
- Distinct markets with no synergy
- Acquired products
- Different brand positioning

**Naming implications**:
- Each name must stand alone
- No parent brand connection needed

### 2. Branded House

One master brand with sub-brands.

**Examples**:
- Microsoft Office, Microsoft Azure
- Amazon Prime, Amazon Web Services
- Salesforce Sales Cloud, Salesforce Service Cloud

**When to use**:
- Strong parent brand
- Related products/services
- Ecosystem strategy

**Naming implications**:
- Name must support sub-branding
- Flexibility for additions
- Parent brand carries trust

### 3. Hybrid

Mix of branded house and house of brands.

**Examples**:
- Facebook (Meta) + Instagram, WhatsApp
- Google + Nest, Fitbit

**When to use**:
- Balance of autonomy and leverage
- Strategic acquisitions
- Market-specific positioning

**Naming implications**:
- Core name for leverage
- Acquired names for recognition
- Clear hierarchy

## Evaluation Framework

### Questions for Each Phase

**Phase 1 Fit**:
- Does the name clearly communicate the initial product?
- Is the positioning clear?
- Does it resonate with target market?

**Phase 2 Fit**:
- Can the name expand to adjacent features?
- Is there room for feature additions?
- Does it limit or enable expansion?

**Phase 3 Fit**:
- Can the name support a platform positioning?
- Does it scale to ecosystem thinking?
- Is there partner integration potential?

### Scoring Rubric

| Score | Phase 1 | Phase 2 | Phase 3 |
|-------|---------|---------|---------|
| 5 | ✅ | ✅ | ✅ |

## Common Pitfalls

### Overly Specific Names

**Problem**: "GDPR" in the name limits expansion

**Examples to avoid**:
- GDPRTracker
- SubProcessorPro
- CookieConsent

**Mitigation**: Abstract beyond initial use case

### Too Abstract Names

**Problem**: No connection to product value

**Examples to avoid**:
- Random invented words with no resonance
- Names that require significant explanation

**Mitigation**: Balance abstraction with clarity

### Platform-Locking Names

**Problem**: Name implies single use case

**Examples to avoid**:
- "Trust" + "GDPR" = only compliance?
- "Attest" + "AI" = only verification?

**Mitigation**: Test expansion scenarios

## The Verb Test

Can the name be used as a verb?

**Good**:
- "I'll Google that"
- "I'll Attest to the data" (works)
- "I'll Vouch for this" (works)

**Bad**:
- "I'll SubProcess this" (doesn't work)
- "I'll ComplyDog this" (doesn't work)

**Why it matters**: Verbs indicate adoption and cultural penetration.

## Documentation

For each candidate, document:

1. **Phase 1 fit**: Initial positioning clarity
2. **Phase 2 potential**: Expansion capability
3. **Phase 3 scalability**: Platform potential
4. **Verb potential**: Can it be used as a verb?
